# Use SvelteKit with adapter-static

The frontend is a single SvelteKit application built with `@sveltejs/adapter-static`. Each Tauri window (Composer, Dock, Inbox, future Settings) loads the same built bundle and points at a different route. This supersedes the earlier slicing-time recommendation of "plain Svelte + Vite, NOT SvelteKit", which was sized for a two-window v0.1 and undervalued future growth. As Inbox accretes sub-views (detail, search) and Settings grows tabbed sections, SvelteKit's filesystem routing, `$app` utilities, and shared layout make each new screen cheaper than rolling a hand router on top of plain Svelte. Cost: a small amount of Kit-specific ceremony (`+page.svelte`, `svelte.config.js`, `svelte-kit sync` in the type-check step) and prerender configuration in the adapter. This is the canonical Tauri + Svelte pattern, so future readers should not be surprised; this ADR exists because the original PRD and slice notes said the opposite and now agree with this decision.
