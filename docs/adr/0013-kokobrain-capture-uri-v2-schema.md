# Kokobrain capture deep-link moves to a typed v2 schema

The `kokobrain://capture` URI emitted by quick-capture moves from the v1 shape — a single opaque `content` parameter holding a pre-rendered markdown body plus `tags` and an optional `title` — to a v2 shape where every capture field travels as its own query parameter. The brain side now owns markdown rendering end to end. The v2 URI always carries `v=2`, `kind`, `vault`, and `captured_at`, plus `tags` when the merged list is non-empty. Per-kind required fields are `text` (for `Note` and `Clip`), `url` (for `Link`), and `path` (for `Shot` and `File`). `Link` also carries an optional `title` resolved from `source_title` then `payload.title` and omits the param entirely when neither is populated. `Shot` and `File` add `mime` when the payload carries one; `File` also adds `original_name` so the brain can preserve the user-facing filename. `path` resolves from `payload.blob_path` (pasted-bytes Shots, app-owned, stable) when present and falls back to `payload.source_path` (drag-in Shots, all Files); the emitter does not file:// encode — the brain side resolves the raw filesystem string. Source context — `source_app`, `source_title`, `source_url` — rides along whenever the Capture has it, with the deliberate exception that `Link` strips `source_title` and `source_url` because they duplicate the canonical `url` / `title` in the payload.

The motivating context is that v1 forced quick-capture to invent the brain's note format. A `Clip` from Chrome had a page title and source URL on the Capture row but no way to express them in the URI — the body was the highlighted text and there was nowhere for the brain to attach a citation footer or YAML metadata without parsing back through the markdown the emitter had already produced. Adding a `title` param for `Link` in v1 (commit 0ea855c) made the asymmetry obvious: `Link` got structured metadata, every other kind was stuck inlining everything into `content`. v2 pushes structure all the way through so the brain can render once with full information instead of re-deriving fields from markdown.

A coexistence window between v1 and v2 on the emitter side was rejected. quick-capture and the brain are deployed together on a single machine by a single user; running a v1 quick-capture against a v2 brain (or vice versa) is a self-inflicted state, not a multi-tenant migration problem. The brain parser will reject v1 URIs after its own v2 release, and the rollout order is: brain v2 ships and tags first, quick-capture v2 ships against that tagged version. A "carry both v1 and v2 fields in one URI" middle path was rejected because the brain v2 parser will not look at `content=` at all, so the redundancy would be dead bytes. Renaming `content` to a per-kind field while keeping the v1 shape was also rejected because the source-context fields have nowhere to live without a discriminator (`kind`), and once `kind` is in the URI the parser may as well branch on schema version.

Schema-wise the change is contained to `src-tauri/src/kokobrain/mod.rs`. `build_capture_uri` now appends pairs in a fixed order (`v`, `kind`, `vault`, kind-specific fields, source fields, `captured_at`, `tags`); `content_for_capture` and the Link-only `title_for_capture` helpers were deleted in favor of `link_title` (the title-fallback chain for Link), `capture_kind_param` (the enum-to-wire-name mapping), and `non_blank` (trim-and-filter for optional source fields). `MalformedPayload("text")` fires for Note/Clip without payload text, `MalformedPayload("url")` for Link without payload URL, and `MalformedPayload("path")` for Shot/File when neither `blob_path` nor `source_path` is present. `BuildError::UnsupportedKind` is gone — every kind now produces a URI. The `tags` merge logic (destination name kebab + user tags, deduped) is unchanged and the empty-merge-omits-param behavior carries over. The picker no longer disables any rows on capture-kind grounds; the ADR-0012 disable rule has been retired.

A revisit added `Shot` / `File` emit (2026-05-19): the picker rule was retired and `path` now flows through. The tradeoff accepted is that `source_path`-backed rows (drag-in Shots, all Files) point at the user's filesystem; if the user moves or deletes the file after routing, the brain's link breaks. Stabilising those by copying into the app's blobs dir at capture time is a separate change; the lean wire-up was preferred so the feature gap closed in one slice.

The cost is one breaking change to the brain parser (which is the point), a test rewrite that replaces v1 `content=` assertions with v2 per-param assertions, and a coordinated release pair. The reward is that adding a new capture-kind field (a clip's quote citation, a link's archived snapshot URL, a shot's path) is now a one-line append at the emitter and a one-line read at the parser, instead of a markdown-format negotiation across two repos.
