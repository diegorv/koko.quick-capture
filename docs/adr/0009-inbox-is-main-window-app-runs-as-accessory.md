# Inbox is the main window; the app runs as macOS Accessory

The Inbox is now the application's main window — the top-level Tauri window declared in `tauri.conf.json`, the one labelled `inbox`. Future product surfaces (Settings, search, "Recently Deleted", etc.) live as SvelteKit routes inside this same window. The Composer (Raycast-style popover summoned by a global shortcut) and the Dock (always-on-top bottom-left widget) remain separate Tauri windows because their UX requires properties — pop-up-over-any-app and persistent-floating — that a single main window cannot deliver.

The activation policy is `Accessory` on macOS: no Dock icon, no system menu bar, only the Tray icon in the menubar. The app is a utility, not a "full" app.

The previous arrangement — one Tauri window per product surface, with `main` pointing at `/composer` and a separate `inbox` window — was rejected after four bug-fix commits in a row (`9097c3a`, `db00761`, `28edab0`, `17680db`) all traced back to the multi-window-per-route shape. Each new feature window re-discovered the same quirks: Tauri's default close-button destroys the WebView; Svelte components only mount once so hidden / shown windows do not refocus; each window's CSS context needs an independent reset; cross-window state synchronisation must marshal through the Tauri event bus with strict name validation. Formalising those quirks into a shared `ManagedWindow` helper was considered but rejected because every future product screen would still be a separate window paying the same fixed cost; the bug surface grows with feature count, not with refactor size. A pure single-window design was also rejected because the Composer's product identity is a popover that spawns over any app — collapsing it into the main window would force the user to activate quick-capture (a visible jank and a clear product regression) before every Note Capture.

The chosen shape — one app shell window (Inbox + future routes), one popover window (Composer), one widget window (Dock) — caps the OS-coupled multi-window surface at exactly two windows. Adding Settings, search, or any future browsing screen does not introduce a new Tauri window; it adds a SvelteKit route inside `inbox`. State sharing between routes uses normal Svelte stores rather than the Tauri event bus; only Composer ↔ shell and Dock ↔ shell need event bus traffic.

The cost of this ADR is a one-commit relabel of the existing windows (`main` -> `composer`, the slice-02 `inbox` builder collapses into the `tauri.conf.json` declaration) and a small capabilities update. The reward is that every future screen is a Svelte route, not a Tauri window.
