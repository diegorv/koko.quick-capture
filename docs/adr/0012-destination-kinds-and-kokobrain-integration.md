# Destinations gain a kind; first integration is kokobrain via the capture deep link

A Destination now carries a `kind` field (closed set: `label`, `kokobrain`) and an opaque `config` JSON blob. The existing label-only behavior is `kind = 'label'` with `config = NULL`, and every pre-existing row migrates to that shape so no destination changes meaning. A `kind = 'kokobrain'` destination additionally carries `config = { "vault": string }`, and routing a Capture to it does what label-routing already does (set `destination_id` + `routed_at`, leave the Inbox, surface in Archive) and **also** fires a `kokobrain://capture?vault=<vault>&content=<payload>&tags=<destName-kebab>` deep link via `tauri-plugin-opener`. The brain app handles the rest: vault resolution, vault switch if needed, quick-note path, file write. quick-capture never learns where the note landed inside the vault; that knowledge lives in brain's quick-note settings (folder, filename format, template).

This slot was reserved by ADR-0010: "integrations slot in later as additional config on the Destination row without changing this lifecycle." ADR-0012 takes that slot at the cheapest viable point. The Routed lifecycle is unchanged; routing semantics remain metadata-only on the Capture; re-route and un-route do not propagate to brain (kokobrain handoff is one-shot side effect, not a synced mirror). Payload mapping is per Capture kind: `Note` and `Clip` send raw text, `Link` sends `[title](url)`, and `Shot` / `File` send a filesystem `path` (see ADR-0013 for the v2 schema and the 2026-05-19 follow-up that wired Shot/File through). Tags are not user-configurable yet; the destination's name is auto-passed as a single tag in kebab-cased form so the brain side can filter by origin without cross-vault casing drift.

A "Router" table separate from Destination was rejected because the GTD lifecycle already lives on captures+destinations and a second table would drift. A per-destination `mode` + `targetFile` config (using `kokobrain://new?silent=true&append=true`) was rejected for v1 because it doubles config surface area and forces quick-capture to invent vault file paths it has no business owning. A brain-side IPC/HTTP endpoint that returns success was rejected because it requires changes on both sides and brain not running is the common case; fire-and-forget via the OS URL scheme matches the disruptive-by-design routing UX. Routing failure detection is deferred to a future slice: if the OS cannot dispatch the URL (`tauri-plugin-opener` errors), the route is aborted and the Capture stays in the Inbox; otherwise the route is optimistic and the user's visual confirmation is brain opening.

Schema-wise the change is two nullable columns on the existing `destinations` table — `kind TEXT NOT NULL DEFAULT 'label'` and `config TEXT` — added via guarded `ALTER TABLE` so existing installs migrate in place and fresh installs get the columns from the inline `CREATE TABLE`. The `tauri-plugin-opener` dependency is scoped to the `kokobrain://` URL scheme via the capability allowlist; broader URL launching would require an explicit capability grant per ADR-0004. Field-level validation for `kind = 'kokobrain'` (i.e. `config.vault` must be a non-blank string) lives in the store layer so the picker and the routing command share one source of truth.

The cost is: one ALTER TABLE pair, one new dep (`tauri-plugin-opener`) with a capability grant, one new Rust command (`route_to_kokobrain`), a kind toggle + vault input in the destination form, and a payload-mapper module. The reward is the first real handoff out of the Inbox - quick-capture stops being a terminal store and becomes a router into the user's notes vault.
