# macOS Spaces, window collection behavior, and focus handoff

Every Tauri window in the app picks an explicit `NSWindowCollectionBehavior` policy on macOS, and the show/dismiss paths defensively work around an AppKit race that surfaces when a window summons onto a Space different than the one it last lived on. The previous shape — windows with AppKit's default `Managed` behavior, a `visible_on_all_workspaces(true)` on the Dock, and a synchronous `set_focus` right after `show` — produced four user-visible bugs that all looked unrelated but traced back to Spaces semantics: shortcuts dragging the user to the window's home Space instead of summoning the window to the user, the Dock disappearing on every other Space, Composer/Inbox/Settings opening visible-but-not-key after a Space transition, and `dismiss_composer` yanking a Space-1 Inbox into Space 2 after a save.

Per-window collection behavior is now explicit and bypasses Tauri's `visible_on_all_workspaces` builder where the builder's single-bit semantics are too coarse:

- **Composer / Inbox / Settings** set `NSWindowCollectionBehaviorMoveToActiveSpace` (`1 << 1`). The window follows the user to whichever Space is active when `show()` runs, instead of forcing the user back to the window's home Space. This is the right shape for any transient app surface that should only exist on one Space at a time. The bit is applied directly via `setCollectionBehavior:` on the underlying `NSWindow`; Tauri's builder does not expose it.
- **Dock** sets `NSWindowCollectionBehaviorCanJoinAllSpaces | Stationary | IgnoresCycle`. The builder's `visible_on_all_workspaces(true)` only sets `CanJoinAllSpaces`, which on recent macOS versions is not enough on its own — the window can drop from view during Mission Control / Space transitions until the next show(). `Stationary` keeps it pinned during the transition; `IgnoresCycle` keeps it out of Cmd+\` window cycling. `MoveToActiveSpace` is stripped defensively because AppKit treats it as mutually exclusive with `CanJoinAllSpaces`.

The Dock's "hide on fullscreen" behavior previously toggled `Entered` / `Exited` on every `NSWorkspaceActiveSpaceDidChangeNotification`. That toggle drifted out of sync as soon as the user swiped between two regular desktops, so the Dock disappeared on every other Space. The observer now uses `CGSSpaceGetType` (a private CoreGraphics SkyLight function used widely by menu-bar utilities — Rectangle, Bartender, AltTab — since macOS 10.11) to ask whether the currently active Space is a fullscreen-tile space, caches the last reported state, and emits a transition only on a real desktop ↔ fullscreen-tile change. CGS is private but stable; using `NSScreen.visibleFrame` as a proxy was rejected because the menu bar reservation reports the same value in fullscreen and desktop modes when the user auto-hides the menu bar.

Cross-Space summoning exposes a timing race between `MoveToActiveSpace` and Tauri's `set_focus` (`makeKeyAndOrderFront:`). The synchronous `set_focus` immediately after `show()` can fire before the move animation completes, leaving the window visible-but-not-key (gray traffic lights, keyboard input goes nowhere) until the user clicks. `show_composer`, `show_inbox`, `show_archive`, and `show_settings` therefore schedule two follow-up `set_focus` calls at 150ms and 280ms on the main run loop. They are idempotent — once the window is key the re-fires are AppKit-level no-ops — and the delays are wide enough to cover the long tail of the animation under load. A single 80ms retry was tried first and missed often enough that the symptom resurfaced; an `NSWindowDidBecomeKeyNotification` observer was considered but rejected as heavier than the race justifies.

`dismiss_composer` used to refocus the Inbox whenever `is_visible()` returned true. With `MoveToActiveSpace` in play, an Inbox shown on Space 1 stays `is_visible()` while the user is on Space 2, and refocusing it from there yanked it into Space 2 — a surprise the user did not ask for. The dismiss path now refocuses the Inbox only when it is BOTH visible AND on the currently active Space, checked via `NSWindow.isOnActiveSpace` (a public AppKit property since 10.6). When the Inbox is not on the active Space, the dismiss falls through to `activate_prev_app`, which restores whichever app the user was actually using on the current Space.

The cost of this ADR is one helper per concern (`apply_move_to_active_space`, `apply_pin_to_all_spaces`, `window_on_active_space`, `schedule_refocus_after_space_move`), one private-API binding (`CGSSpaceGetType`), and a ~430ms window on cross-Space summon during which focus may not yet have settled. The reward is that none of the four pre-fix Spaces-related bugs can resurface without an explicit change to the helpers — the policies are pinned at build time and the dismiss path is guarded.
