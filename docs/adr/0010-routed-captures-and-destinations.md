# Captures gain a Routed state; Inbox and Archive are separate views with separate search

A Capture now has a third terminal-ish flag, `Routed`, alongside the existing `Starred` and `Deleted`. Routing means the user assigned the Capture a `Destination` (a small user-managed label like "Todoist" or "Readwise") to indicate "I have decided what to do with this." A Routed Capture leaves the [Inbox](../../CONTEXT.md#inbox) view and surfaces only in the [Archive](../../CONTEXT.md#archive) view. The Inbox and Archive share the main window but are distinct surfaces with **separate search indexes / search scopes** — typing into the Inbox search box never returns Routed Captures, and typing into the Archive search box never returns un-Routed Captures. This is the GTD layer of the product: the Inbox is the queue of things still owing a decision; the Archive is the receipt of decisions already made.

The schema change is two nullable columns on `captures` (`destination_id` referencing a new `destinations` table, plus `routed_at`) and a `destinations` table with `id / name / color / created_at / deleted_at`. Routing is reversible — the user can re-route a Capture to a different Destination or un-route it back to the Inbox; both are metadata-only mutations that do not touch payload, so they sit inside the spirit of ADR-0003 (the kind and payload still never change). A Destination is soft-deleted rather than removed: orphan Captures keep their reference and surface in the Archive with a "(deleted)" marker so the user can re-route or restore. Names are unique only among live Destinations, enforced by a partial unique index.

Several alternatives were rejected. A multi-destination model (one Capture, many Destinations) was rejected because it muddies the "Routed" boolean — partially-routed Captures would need a third state for the Inbox/Archive split — and because the GTD habit being supported is "one next action per item." Making Trash a built-in Destination was rejected because deletion means "I should not have captured this" and routing means "I dealt with this"; collapsing them loses a signal the user cares about. Using the existing tag/label mental model (i.e. a many-to-many tagging table without a Routed boolean) was rejected because tags do not move items out of the Inbox — the Inbox would keep growing forever and the GTD signal would be lost. Building integrations now (a Destination = a real handoff to Todoist / Readwise APIs) was deferred because v1 only needs the label to break the "stuff piles up forever" loop; integrations slot in later as additional config on the Destination row without changing this lifecycle. A unified Inbox+Archive search was rejected on the same grounds as the view split: GTD relies on the Inbox surface being the *only* thing that demands attention, so search results from already-processed Captures would dilute that signal — when the user is searching the Archive they have already context-switched into "what did I do with X?" mode and merging the two would slow both flows.

The Archive lives inside the main window as a sibling view to the Inbox, reached via a top-of-window switcher and a global shortcut. Triage is keyboard-driven (`R` opens the Destination picker on the selected Inbox row; the picker auto-opens in create mode when no Destinations exist yet, so a brand-new install can start triaging in one keystroke). Bulk routing is deliberately not in v1 — single-row triage is the muscle memory being built. Star persists across routing because it is an orthogonal "I care" signal, not a triage state.

The cost of this ADR is a new table, two new columns, a new SvelteKit route, a Destinations section in Settings, and the picker UI. The reward is that the Inbox has a clear exit door, the Archive has a clear entry, and the product stops being a place where copied things go to be forgotten.
