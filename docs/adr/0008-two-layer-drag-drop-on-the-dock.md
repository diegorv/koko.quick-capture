# Two-layer drag-drop on the Dock

The Dock accepts drops of files, URLs, plain text, and image bytes (see CONTEXT.md). macOS surfaces these as a single drag event, but the Tauri runtime exposes them through two different APIs that each cover only part of the surface: the native `WebviewWindow::on_drag_drop_event` handler delivers file paths but not text or `text/uri-list` payloads, and the WebView's HTML5 `ondrop` handler delivers everything in `DataTransfer` *except* file paths from a Finder drag (those arrive only via the native channel on macOS). We use both layers in parallel rather than choosing one. The Tauri layer is the source of truth for file drops; the HTML5 layer is the source of truth for text, URLs, and image bytes. When a drop arrives via both layers (which happens for Finder file drags that the WebView also sees), we de-duplicate by preferring the Tauri payload and ignoring the HTML5 event for that drop. A single-layer alternative was rejected because each layer alone leaves one of the four documented drop types unreachable. Routing both layers through a pure `drag_drop::decide_drop(payload) -> Vec<CaptureInput>` function keeps the OS coupling at the seam and lets the decision logic stay unit-testable.
